# Codex 登录拆分为独立 CLI 命令，并让 UI 通过 backend 调用该命令

## Goal

把当前埋在 `CodexAccountService.start_device_login()` 里的 Codex 登录触发逻辑，重构成一个明确、可独立调用、可测试的 CLI 子命令；同时让 UI 不再直接依赖 service 内部登录实现，而是继续通过 backend 的 D-Bus 接口，由 backend 异步调用这个 CLI 命令。

这次整理后的目标不是“表面拆命令”，而是把以下几件事一次理顺：

1. 登录生命周期由独立 CLI 命令完整负责
2. backend 调用 CLI 时不阻塞 D-Bus / GTK
3. 登录中状态可跨进程共享
4. 账号 manifest / auth 文件在多进程下仍然安全
5. 默认 label 优先从 token 本地解析邮箱，不依赖额外 OpenAI API

## Key conclusions after review

### 结论 1：CLI 命令必须负责完整登录生命周期
不能做成“只拉起 terminal 然后自己退出”的半成品命令。

真正靠谱的 `codex-login` 命令必须自己负责：
- 检查是否已有登录进行中
- 备份当前 auth
- 拉起终端执行 `codex login`
- 等待登录完成
- 检查新凭据是否有效
- 导入账号快照
- 登录失败时恢复旧 auth
- 清理状态文件和临时文件

否则所谓“拆出去的 CLI”只是换了个壳，核心耦合还在，没意义。

### 结论 2：backend 必须异步调用 CLI
backend 不能 `subprocess.run()` 阻塞等待 CLI 完成。

因为登录过程可能要等用户交互数分钟：
- D-Bus 会卡住
- GTK 客户端可能超时
- UI 体验会很烂

正确方式是：
- backend 用异步子进程启动 CLI
- 立即返回 `started=True`
- 再在后台监听 CLI 退出
- CLI 退出后刷新 Codex account 状态并发信号

### 结论 3：必须补跨进程文件锁
当前 `CodexAccountService` 用的是 `threading.Lock()`。

这玩意只对单进程线程有用，对多进程完全不够。
一旦 backend daemon 和独立 CLI 进程并发修改：
- `accounts.json`
- `~/.codex/auth.json`
- account snapshot 文件
就有机会写坏。

所以这次设计必须加入跨进程文件锁，例如：
- 在 `codex_accounts_dir` 下单独维护 lock file
- 用 `fcntl.flock()` 保护 manifest/auth/snapshot 的读写临界区

## Scope

本次计划的目标是重构登录触发链路，不改动已有账号切换/重命名/删除的业务语义。

明确在 scope 内：
- 新增 CLI 子命令 `linux-agent-island codex-login`
- backend 改为异步调用该 CLI
- 登录状态改为跨进程可见
- 账号文件操作加跨进程锁
- 默认 label 解析规则明确化
- 更新测试与文档

明确不在第一版 scope 内：
- frontend 直接自己 subprocess 调 CLI
- 新增单独二进制命令名（例如 `linux-agent-island-codex-login`）
- 在线调用 OpenAI API 查询邮箱作为主要 label 来源
- 改造整个 Codex account 管理架构

## Proposed architecture

## 1. 保留 UI -> backend D-Bus 入口
前端调用关系保持不变：
- UI 按钮
- `frontend_client`
- backend D-Bus `StartCodexDeviceLogin`

这样改动面最小，也不会把 GTK 层搞成 subprocess 垃圾堆。

## 2. backend 内部改为异步调用独立 CLI
`StartCodexDeviceLogin` 的实现改为：

1. 基础校验
2. 启动子进程：
   - `[sys.executable, "-m", "linux_agent_island.cli", "codex-login", "--label", ...]`
3. 立即返回 `started=True`
4. 用后台 watcher 等待 CLI 退出
5. CLI 结束后：
   - 刷新 account 状态
   - 发 `CodexAccountsChanged`
   - 如有必要补日志 / telemetry

注意：
- 优先用 `sys.executable -m linux_agent_island.cli`
- 不依赖 PATH
- 不依赖 entrypoint 是否正确安装到 systemd 的运行环境里

## 3. 独立 CLI 负责完整生命周期
新增 CLI 子命令：
- `linux-agent-island codex-login [--label <name>]`

其职责应完整封装登录流程：
- 获取配置
- 获取跨进程文件锁
- 检查是否已有登录进行中
- 创建登录状态文件
- 备份并必要时导入旧 auth
- 启动 terminal 执行 `codex login`
- wait 登录完成
- 读取新 auth
- 导入新账号
- 成功时清理备份
- 失败时恢复旧 auth
- 最终删除状态文件并返回 exit code

## 4. 登录中状态改为文件真相源
不要再把 `_login_in_progress` 当真相。

改造目标：
- `device_login_in_progress` 由共享状态文件推导
- backend / frontend / CLI 看到的是同一事实源
- 即使登录从独立 CLI 发起，daemon 也能知道“现在正在登录”

可以保留内存标志作为本进程缓存，但不能再依赖它决定业务状态。

## 5. label 解析采用本地 token 优先，不依赖额外 API
默认 label 规则整理为：

1. 用户显式传入 `--label` 时，用显式 label
2. 否则尝试本地解析 `id_token.email`
3. 否则尝试本地解析 `access_token.email`
4. 否则尝试本地解析 `access_token["https://api.openai.com/profile"].email`
5. 再 fallback 到：
   - `Current account`
   - 或 `Codex account N`

理由：
- 现有 auth.json 已能本地拿到邮箱
- 少一层网络依赖
- 少一层失败面
- 不需要假设存在稳定公开的“按 access token 查邮箱” OpenAI API

## Design decisions

### Decision A: 子命令名称
采用：
- `linux-agent-island codex-login`

不建议第一版新增单独 script 名。
原因：
- 已有 CLI 结构就是 subcommand 模式
- 简洁
- 不污染命名空间

### Decision B: backend 是否直接调用 service 还是调用 CLI
采用：
- backend 调 CLI

原因：
- 满足“登录独立出来，作为 cli 命令”这一目标
- 手工调试路径和 UI 路径统一
- 避免存在两套登录触发实现

### Decision C: CLI 是否阻塞
采用：
- CLI 自己是阻塞式完整生命周期命令
- backend 对 CLI 的调用是异步的

这是最重要的整理点：
- CLI 负责完整流程
- backend 对外不阻塞

两边各司其职，不拧巴。

### Decision D: 是否调用 OpenAI API 查邮箱
采用：
- 不作为主方案
- 优先本地解析 token

如未来确实遇到 token 不含邮箱、又明确有稳定接口可查，再评估作为可选 fallback。

## Step-by-step plan

### Step 1: 盘清并拆分 `CodexAccountService` 的职责边界
目标：把“状态管理”和“登录 orchestration”拆清楚。

需要识别并整理以下职责：
- auth payload 读取与凭据判定
- manifest 读写
- snapshot 读写
- label 解析
- 登录状态文件管理
- auth 备份 / 恢复 / 清理
- terminal launch command 生成
- wait 登录完成
- 导入当前 auth 为新 account

预期结果：
- service 中保留账户领域逻辑与文件操作能力
- 登录生命周期 orchestration 可被 CLI handler 明确调用

### Step 2: 引入跨进程文件锁
修改 `linux_agent_island/codex_accounts.py` 或抽出独立 lock 工具模块。

要求：
- 所有 manifest / auth / snapshot 的关键读写放到同一把跨进程锁保护下
- 推荐 lock file 位置：
  - `config.codex_accounts_dir / ".accounts.lock"`
- 推荐实现：
  - `fcntl.flock()`
- 需要明确锁的作用范围：
  - 读取 manifest 并修改后回写
  - auth 备份/恢复
  - snapshot 写入
  - 登录完成导入

同时：
- 保留 `threading.Lock()` 也行，但只能作为进程内补充
- 真正的互斥必须由 file lock 保证

### Step 3: 把“登录中状态”改成跨进程可见
整理当前状态文件逻辑，避免仅靠随机 status file + 内存布尔值。

重点：
- `get_status()` 应能判断是否存在有效登录流程进行中
- CLI 发起登录时要写入共享状态
- CLI 完成/失败时清理共享状态
- backend 在 CLI 运行期间查询状态时能看到 `device_login_in_progress=True`

这里要注意：
- 状态文件命名和生命周期要稳定
- 不要只生成 daemon 自己知道的随机临时状态文件，然后别的进程根本找不到

### Step 4: 定义 CLI 子命令接口
修改 `linux_agent_island/cli.py`：

新增：
- `codex-login`

参数：
- `--label <name>`

- 创建 `AppConfig.default()`
- 初始化登录所需 service / helper
- 执行完整登录流程
- 输出必要日志和错误
- 返回 exit code

建议 exit code：
- `0`: 登录流程完成且成功导入账号
- `1`: 已有登录进行中 / 用户取消 / 登录失败 / 环境错误

如果后续需要区分更细错误码，再细分；第一版不用装复杂。

### Step 5: 让 CLI 成为唯一登录执行路径
核心要求：
- 不再保留另一套 backend 内部直接执行登录逻辑

这一步要确保：
- 不出现“旧 service 路径还能偷偷直接发起登录”

## Risks and tradeoffs

### 风险 1：跨进程状态设计不严谨
如果状态文件只是临时拼凑，daemon 看不到、或崩溃后残留，就会导致 UI 卡在假登录中。

### 风险 2：文件锁范围不对
锁太小会漏保护，锁太大可能拖慢，但比起写坏数据，性能这点代价根本不值一提。

### 风险 3：CLI 与 backend 责任边界重新混乱
如果后面又偷懒把一半逻辑放回 backend，最终还是两套实现并存，会继续烂。

### 风险 4：started 语义混淆
必须明确：
- backend 返回 `started=True` = CLI 成功启动
- 不等于用户已经登录成功

### 风险 5：过度设计
第一版不要引入多余能力：
- 不新增独立 script 名
- 不直接从 frontend 自己 subprocess
- 不先搞联网查邮箱 fallback

## Open questions

1. `codex-login` 的 exit code 是否需要区分：
   - 用户取消
   - 启动失败
   - 登录失败
   第一版可以先统一非零，后续再细分。

2. 登录状态文件应该采用固定文件名还是 session-specific 文件名 + 全局索引？
   倾向固定可发现入口，别为了“优雅”把可观察性搞丢。

3. 如果已有登录中，backend D-Bus 是否返回 started=False 还是 DBus error？
   倾向 started=False 或明确可识别的业务错误，别让 UI 难处理。

## Recommended implementation order

1. 先理顺跨进程文件锁
2. 再理顺共享登录状态文件
3. 再定义并实现 `codex-login` 完整生命周期命令
4. 再让 backend 改成异步调用 CLI
5. 再检查 UI 文案和按钮状态
6. 最后补测试和文档

这个顺序比之前稳得多。
先把并发和状态真相源钉死，再拆调用链，不然你只是换了个入口，坑一个没少。
